# AI 昏君版改造规划 v1

## 1. 文档目标

本文档用于沉淀 `boluobobo-ai-court-tutorial` 从「勤政版 AI 朝廷」改造成「昏君工作流」版本的完整规划，方便后续按文档拆分任务、逐批执行和验收。

本次改造目标不是简单换皮，而是把产品默认工作方式改成：

- 用户只提目标，不盯过程
- 内廷自动过滤噪音、拆任务、压缩汇报
- 六部自主推进，非必要不打扰
- 后宫/生活机构负责轻娱乐、陪伴、日常服务
- 外部技能体系负责生图、发文、小说、音乐、视频等内容生产

## 2. 产品定位

### 2.1 新版本定义

建议命名：

- 主名：`AI 昏君`
- 副标题：`你只管摆烂，群臣自己卷起来`

### 2.2 核心价值

- `真省心`：默认少打扰用户，只呈现结论、待批项、风险。
- `真能用`：内廷自动拆解任务，六部和司局自己完成执行。
- `轻娱乐`：支持生图、娱乐文、社媒文、歌曲、视频、小说等内容生产。
- `可扩展`：通过外部 Skills 和 API 继续扩容，不把所有能力硬写死在主项目内。

### 2.3 V1 用户体验原则

- 默认只有一个入口：`掌印总管`
- 默认只有三类汇报：`已办妥`、`待拍板`、`有风险`
- 默认不要求用户选部门
- 默认不让用户扮演项目经理
- 默认允许用户在工作流中穿插娱乐与生活需求

## 3. 目标架构

### 3.1 组织结构

#### 内廷

| Agent ID | 展示名 | 作用 |
|---|---|---|
| `main` | 掌印总管 | 唯一默认入口，任务分诊，压缩汇报，少打扰 |
| `qijuzhu` | 起居注官 | 摘要、日报、周报、事项归档 |
| `tingyi` | 廷议官 | 方案比较、待批事项整理、供拍板 |

#### 六部

| Agent ID | 展示名 | 作用 |
|---|---|---|

#### 后宫 / 生活机构

### 3.2 工作流总原则

所有需求统一先到 `main`：

1. 识别需求类型
2. 能直接答就直接答
3. 需要执行就自动分发
4. 执行结果先内部汇总
5. 只把关键结果上呈给用户

### 3.3 打扰规则

只有以下情况允许主动打扰用户：

- 需要花钱
- 需要发布
- 存在风险边界
- 多方案需要拍板
- 权限或凭证缺失

其余情况原则上：

- 先做
- 再报
- 只报结论

## 4. 外部能力接入规划

## 4.1 内容生产外挂总览

V1 建议不把所有外部能力深度内嵌在主项目后端逻辑中，而是采用：

- `文档明确接入`
- `安装后提示接入`
- `Agent Prompt 约定优先调用`
- `后续再按需要做深度集成`

这样能保证第一阶段改造风险较低，同时保留扩展性。

## 4.2 `baoyu-skills` 接入

适合接入的 Agent：

- `huagong`
- `jiaofangsi`
- `libu`

适合承接的能力：

- 生图
- 海报
- 娱乐配图
- X 文
- 微博文
- 公众号图文
- 封面图
- 轻量内容包装

接入策略：

- 在 README 中单列“推荐外挂能力”
- 在安装脚本最后提示推荐安装
- 在相关 Agent 的 `identity.theme` 中写明“优先调用相应 Skills”
- GUI 第一版只提供语义入口，不对外部仓库做强耦合绑定

## 4.3 `webnovel-writer` 接入

适合接入的 Agent：

- `hanlinyuan`

适合承接的能力：

- 长篇小说项目初始化
- 卷纲与章节规划
- 正文创作
- 审稿与追读力检查
- 小说项目可视化浏览

接入策略：

- 第一版把 `hanlinyuan` 定义成“小说专属工作站”
- 文档中明确使用方式
- Prompt 里写明优先走结构化小说工作流
- 暂不强行把小说 Dashboard 混入当前主 GUI

## 4.4 `Suno API` 接入

适合接入的 Agent：

- `jiaofangsi`

建议承接的能力：

- 生歌曲
- 主题曲
- 人设曲
- 小剧场背景音乐
- 宣发视频配乐

建议在角色定位中加入：

- 当用户要“来首歌”“写歌并生成歌曲”“做角色主题曲”时，优先调用 Suno API。
- 生成前自动整理歌曲意图：主题、风格、歌词语言、时长、情绪、用途。
- 用户可直接提供：`歌词`、`歌名`、`风格标签`。
- 系统可额外配置“生成歌词”的系统提示词，以及“整理成 Suno 提交格式”的系统提示词。
- 生成后回传：
  - 作品摘要
  - 风格说明
  - 链接或文件位置
  - 是否适合进一步配视频

### 4.4.1 已明确的接口约定

运行时环境变量：

- `SUNO_API_URL`
- `SUNO_KEY`

已知接口：

- 提交生成：`POST /suno/submit/music`
- 查询任务：`GET /suno/fetch/{task_id}`

建议拼接方式：

- 提交地址：`${SUNO_API_URL}/suno/submit/music`
- 查询地址：`${SUNO_API_URL}/suno/fetch/{task_id}`

认证方式：

- `Authorization: Bearer ${SUNO_KEY}`

### 4.4.2 教坊司的输入整理规则

教坊司收到“生歌”请求时，优先整理为以下字段：

- `title`：歌名
- `tags`：风格标签
- `prompt`：歌词正文
- `mv`：模型版本

优先使用模式：

- 第一阶段默认走“定制模式”
- 即用户提供核心歌词、歌名、风格标签，教坊司负责补齐和格式化

默认流程：

1. 收集用户给出的歌词、歌名、风格标签
2. 若信息不完整，先按系统提示词补生成歌词或补足结构
3. 整理为 Suno 请求体
4. 提交异步任务
5. 轮询查询结果
6. 汇报音频、封面、视频链接及摘要

V1 技术落点建议：

- 文档中正式预留 `SUNO_API_URL`、`SUNO_KEY`
- 安装脚本里预留配置说明，不强制要求填写
- `jiaofangsi` 的 prompt 中明确“当具备 Suno API 配置时优先调用”
- 第一阶段可以把这部分视为“真实外部执行能力”，不是纯概念占位
- 若后续要做 GUI 深集成，可在 `gui/server/index.js` 新增任务触发/记录接口

## 4.5 `SEEDDANCE API` 接入

适合接入的 Agent：

- `jiaofangsi`
- 视情况由 `huagong` 协作生成分镜图或角色图参考

建议承接的能力：

- 生短视频
- 宣传视频
- 角色短片
- 宫廷风娱乐片段
- 文图转视频的轻内容

建议工作流：

1. 用户提出视频需求
2. `jiaofangsi` 先整理分镜/脚本/风格
3. 需要角色参考图时转给 `huagong`
4. 再调用 SEEDDANCE API 出视频
5. 汇报视频主题、时长、版本说明、输出位置

### 4.5.1 已明确的接口约定

运行时环境变量：

- `SEEDDANCE_API_URL`
- `SEEDDANCE_KEY`

已知接口：

- 提交文生视频：`POST /task/jimeng/text2video`
- 查询任务结果：`GET /task/{task_id}`

建议拼接方式：

- 提交地址：`${SEEDDANCE_API_URL}/task/jimeng/text2video`
- 查询地址：`${SEEDDANCE_API_URL}/task/${task_id}`

认证方式：

- `Authorization: Bearer ${SEEDDANCE_KEY}`

### 4.5.2 教坊司的视频生成规则

教坊司收到“生视频”请求时，优先整理为：

- `prompt`：视频提示词
- `aspect_ratio`：画幅比例
- `seed`：随机种子，可选

默认流程：

1. 将用户需求压缩成 150 字以内的视频提示词
2. 明确画幅比例，如 `16:9`、`9:16`
3. 提交异步任务
4. 轮询查询接口直到完成
5. 回传视频链接、版本摘要和后续优化建议

补充说明：

- 当前接口文档显示视频默认是 5 秒时长，因此产品上应定义为“短视频片段能力”，适合宣传片、角色短片、MV 片段和娱乐素材，不适合作为长视频编辑器。

V1 技术落点建议：

- 文档中正式预留 `SEEDDANCE_API_URL`、`SEEDDANCE_KEY`
- 安装脚本加入可选配置提示
- `jiaofangsi` prompt 中约定“短视频、MV、娱乐向视频默认走该 API”
- 第一阶段不在主 GUI 中深做视频编辑能力，但可以预留任务状态展示位

## 9. 风险与边界

### 9.1 风险项

- 改造范围较大，容易只换文案不换工作流
- 前端名称和后端角色映射不一致时会造成体验割裂
- 若第一版就同时深集成 Suno / SeedDance / 小说工作流 / 外部 Skills 的 GUI 生命周期管理，复杂度会明显上升
- 娱乐能力需要合规边界，避免越界内容

### 9.2 控制策略

- 第一版优先改“默认工作流”和“组织结构”
- `baoyu-skills` 和 `webnovel-writer` 优先做软接入
- `Suno` 与 `SeedDance` 因接口已明确，可按“运行时环境变量 + 教坊司执行入口”的方式做轻执行集成
- 图片、娱乐文、视频等内容由 `xingbu` 负责边界审查
- 分批上线，每一批都保证可运行

## 10. 验收标准

完成 V1 后，至少应满足以下标准：

- 安装脚本生成的是新组织结构
- 用户默认只对 `main` 说话即可完成大多数任务
- 首页主信息是“已办妥 / 待拍板 / 风险 / 娱乐入口”
- 页面和后端映射中不再以旧朝廷勤政逻辑为主
- 文档中清楚说明外部技能与 API 的接入方式
- 教坊司具备明确的：
  - 娱乐文能力
  - 生歌能力规划与接口约定
  - 生视频能力规划与接口约定
